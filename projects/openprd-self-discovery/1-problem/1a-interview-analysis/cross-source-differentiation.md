## 1. Discovered Differentiation Axes

### Axis A: Organizational Scale & Team Structure
**Evidence:** All four sources operate at dramatically different scales, which directly shapes their needs.
- Diego is a solo operator who IS "the cross-function" [Source: diego-analysis.md]
- Sarah works solo on discovery but presents to leadership at a 50-person company [Source: sarah-analysis.md]
- Tom works with a UX researcher and reports to a CEO at a 300-person company [Source: tom-analysis.md]
- Aisha leads a 15-person product/design/eng team at a Fortune 500 and reports to a VP [Source: aisha-analysis.md]

### Axis B: Discovery Methodology Preference (Qual-First vs. Data-First)
**Evidence:** Sources split clearly on where discovery starts.
- Diego and Sarah start from qualitative data (interviews/calls) and want to understand "what" users are saying [Source: diego-analysis.md, sarah-analysis.md]
- Tom explicitly starts from quantitative data (funnels, cohorts) and uses qualitative to explain anomalies [Source: tom-analysis.md]
- Aisha uses both but her primary workflow is qualitative (affinity mapping, Dovetail) with analytics as enrichment [Source: aisha-analysis.md]

### Axis C: Output Complexity Tolerance
**Evidence:** Sources vary dramatically in how much output they can consume and how they use it.
- Diego explicitly rejects comprehensive output: "I will never read a 20-page pain report" [Source: diego-analysis.md]
- Sarah values Phase 1 depth but finds Phase 2 too generic [Source: sarah-analysis.md]
- Tom wants MORE depth on the data side but in a different format (metric hooks, experiment briefs) [Source: tom-analysis.md]
- Aisha wants full pipeline depth but in enterprise-formatted templates [Source: aisha-analysis.md]

### Axis D: Technical Comfort Level
**Evidence:** All four happen to be technically capable, but they observe different barriers for their peers.
- Diego is a developer but flags that non-technical founders would "bounce in 5 minutes" [Source: diego-analysis.md]
- Sarah is "comfortable with code" but finds API key setup "intimidating" — "I'm a PM, not a developer" [Source: sarah-analysis.md]
- Tom lives in BigQuery and writes his own queries [Source: tom-analysis.md]
- Aisha is tool-savvy but immediately flags CLI as non-shareable with her team [Source: aisha-analysis.md]

### Axis E: Collaboration Requirements
**Evidence:** Clear gradient from zero collaboration needs to team-as-default.
- Diego: Zero — is the only person in the company [Source: diego-analysis.md]
- Sarah: Some — needs to present to leadership, share with designer [Source: sarah-analysis.md]
- Tom: Moderate — works with UX researcher, needs to feed experiment backlog [Source: tom-analysis.md]
- Aisha: Critical — "A repo on my laptop is a non-starter for team workflows" [Source: aisha-analysis.md]

## 3. Convergences

These patterns appear consistently across sources regardless of position on the axes. These are strong signals.

### C1: Interview/qualitative analysis is the strongest capability
All four sources rate the qualitative analysis as the tool's best feature.
- Diego: "the AI analysis is legit" [Source: diego-analysis.md]
- Sarah: "genuinely impressive," "A-tier" [Source: sarah-analysis.md]
- Tom: "The analysis was good" [Source: tom-analysis.md]
- Aisha: "the best I've seen," "80% aligned with what I'd have done manually" [Source: aisha-analysis.md]

### C2: /pair mode is the most valued interaction pattern
All sources who used /pair found it significantly more valuable than the pipeline.
- Diego: "THAT is what I needed. Direct, opinionated, actionable." [Source: diego-analysis.md]
- Sarah: "worth the whole setup time" — proactive thinking about root causes [Source: sarah-analysis.md]
- Tom: "contextual intelligence is valuable" — marketplace-specific metric frameworks [Source: tom-analysis.md]
- Aisha: Did not explicitly mention /pair, but valued the executive summary (a synthesis output) [Source: aisha-analysis.md] [Inference: pair mode's synthesis quality extends to pipeline outputs]

### C3: Quant-qual synthesis is universally desired but doesn't work automatically
All sources with quantitative data wanted automatic synthesis; none got it.
- Sarah: "two separate tools that happen to live in the same repo" [Source: sarah-analysis.md]
- Tom: "data points side by side" not "true integrated analysis" [Source: tom-analysis.md]
- Aisha: Wanted Salesforce account data overlaid on qualitative findings [Source: aisha-analysis.md]
- Diego: Did not have analytics tools, but the pattern holds for all sources with data [Inference: universal need for users with both data types]

### C4: Export/output format doesn't match real-world delivery needs
All sources found output formats inadequate for their actual delivery context.
- Diego: Needs 1-page brief, gets 20-page report [Source: diego-analysis.md]
- Sarah: Needs Google Slides/PDF, gets HTML [Source: sarah-analysis.md]
- Tom: Needs experiment briefs with metric hooks, gets Jira stories [Source: tom-analysis.md]
- Aisha: Needs PowerPoint in brand template, gets HTML; needs Confluence in company template, gets generic format [Source: aisha-analysis.md]

### C5: Product context onboarding is smooth
All sources found the initial conversational onboarding positive.
- Diego: "conversational, not a form... Cool" [Source: diego-analysis.md]
- Sarah: "That felt smooth" [Source: sarah-analysis.md]
- Tom: Not explicitly commented but no complaints [Source: tom-analysis.md]
- Aisha: Not explicitly commented but no complaints [Source: aisha-analysis.md]

### C6: Git repo / CLI entry point is a recognized adoption barrier
Multiple sources flag this as a structural limitation.
- Sarah: "90% of PMs would never clone a repo" [Source: sarah-analysis.md]
- Diego: "if I were a non-technical founder, I'd bounce in 5 minutes" [Source: diego-analysis.md]
- Aisha: "I can't share this with my design lead" [Source: aisha-analysis.md]
- Tom: Dream tool includes "a UI that isn't a terminal" [Source: tom-analysis.md]

## 4. Divergences

Pain points, needs, or behaviors specific to certain positions on the axes.

### D1: Output volume — too much vs. too little (Axis C divergence)
- **Low tolerance (Diego):** "WAY too much," wants 1-page brief, /decide command [Source: diego-analysis.md]
- **High tolerance, different format (Tom):** Wants MORE output but as metric-connected experiment briefs [Source: tom-analysis.md]
- **High tolerance, enterprise format (Aisha):** Wants full pipeline depth but in company-specific templates [Source: aisha-analysis.md]
- **Moderate tolerance (Sarah):** Phase 1 is right; Phase 2 needs more depth and concreteness [Source: sarah-analysis.md]

**Implication:** Output volume needs to be context-adaptive. One-size-fits-all is wrong for every segment.

### D2: Collaboration needs — solo vs. team (Axis E divergence)
- **No collaboration (Diego):** Never mentions sharing; is the only user [Source: diego-analysis.md]
- **Critical collaboration (Aisha):** "a non-starter for team workflows" — needs URL sharing, role-based access [Source: aisha-analysis.md]
- **Moderate (Sarah, Tom):** Need to present/share but can work around it [Source: sarah-analysis.md, tom-analysis.md]

**Implication:** Collaboration features are table stakes for enterprise but irrelevant for founders.

### D3: Integration depth priorities (Axis B divergence)
- **Quantitative-first (Tom):** BigQuery depth, Statsig experiment history, metric hooks [Source: tom-analysis.md]
- **Qualitative-first (Sarah, Aisha):** Intercom triangulation, Salesforce enrichment, Gong import [Source: sarah-analysis.md, aisha-analysis.md]
- **No integrations (Diego):** Only needs Zoom recording import [Source: diego-analysis.md]

**Implication:** Integration investment should follow segment priority. Quant integrations serve growth PMs; CRM/support integrations serve enterprise PMs; cloud recording import serves founders.

### D4: Enterprise governance needs (Axis A divergence)
- **Specific to enterprise (Aisha):** SOC 2, SSO, data residency, draft-mode Jira, custom Confluence templates, access controls [Source: aisha-analysis.md]
- **Not relevant** for Diego, Sarah, or Tom at their current scale.

**Implication:** Enterprise governance is a separate feature track — critical for enterprise, invisible to others.

### D5: Experiment workflow (Axis B divergence — specific to Growth PM)
- **Specific to Tom:** Experiment briefs, experiment history filtering, minimum detectable effect, test design [Source: tom-analysis.md]
- **Not mentioned** by any other source.

**Implication:** "Growth Discovery" is a distinct workflow that serves a specific audience. Could be a high-value niche.

### D6: Language/localization (Axis A divergence)
- **Specific to Diego:** Portuguese docs mixed with English interface is "confusing" and signals "this wasn't built for me" [Source: diego-analysis.md]
- Other sources did not comment on language.

**Implication:** International adoption requires consistent language. Portuguese-first docs limit global reach.
